1. Як встановити імя та email git репозиторія
git config --global user.name "Alex Tkachenko" 
-config - конфігурація 
-user.name імя автора
--global - налаштування будуть задіяні для всіх репозиторіїв

2. Отримати всі налаштування гіта
git config --list

3. Команди pawerShell
- pwd - показує абсолютний шлях користувача 
- mkdir newfolder - створити папку в дерикторії в якій знаходищся 
- echo "Hello World" > file.txt - команда створення файла (назва та розширення) та що в файлі буде 
- ls - подивитися список файлів в папці
- cat file.txt - прочитати файл
- rm file.txt - видалити файл 
- cd .. - вийти в батьківську папку
- ls -Force - показати всі скриті папки

4. git init - створення репозиторія в папці в якій ми знаходилися. В цій папці створюється скрита папка .git

5. git status - показати файли підготовленні до коміта

В репориторії гіта занходяться об♥єкти гіта назва яких захешировані. Об'єкт гіт це версія файла. Кожен раз коли ми робимо коміт ми створюємо новий об'єкт гіт для файла 

6. git add додати файли в індекс (підготувати до загрузки на репозиторій). Якщо після git add поставити . то ми всі файли перенесемо до індексу. Якщо назву файлу то тільки файл 

7. git commit - команда для переносу підготовлених даних з індексу в репозиторій 

8. git checkout - команда для переносу об'єктів гіта з репозиторія в робочу дерективу

Статуси фалів 
- untracked - файлі які тільки попали в робочу дерективу та ще не відстежуються гітом
- staged - файли які ми додали в індекс 
- unmidified - файли які потрапили в репозиторій 
- modified - файли які раніше були в індексі та репозиторії але були змінені в робочій дерикторії

Типи об'єктів гіт 
- blob - файли
- tree - папки
- commit - об'єкт який створюється коли ми зберігаємо теперішню версію проекту. Це вказівнит на tree
- annotated tag

При формкванні нової версиі проетку гіт спочтаку створює об'єкти blob, tree які вказують на blob а потім commit який вказує на tree корневий. 
Кожен новий коміт має посилання на батьківський коміт

У гіта є head який вказує на коміт і ми можемо змінювати head що б він вказував на інший commit і цей коміт буде відображатися в робочій дерикторії

Гілка в в гіті завжи показує самий актульний коміт в даній гілці

9. git status - показує стан гіта (які файли були зміненіб які файли були додані і тд)

10. git commit -m "this is my first commit" - команда яка створює коміт

11. git log - показує історію комітів та їх хеши 

12. git checkout "commit hash" - повернення до коміта по його хешу

git checkout "branch name" - можна переміщатися на іншу гілку яка вказує на останній коміт в даній гілці

13.  git cat-file -t 130066b5921 - файли в папці об'єкт яка знаходиться в папці .git не можливо прочитати за допомгою команди cat. Назва папки це перші два символи хеш коду об'єкта. git cat-file -t 130066b5921 де -t це тип а після нього це хеш код об'єкта який ми хочемо прочитати. -p це інформація про об'єкт

14. git cat-file -t git cat-file -p ми можемо дізнаватися хеш коди інших об'єктів та відкривати їх 

Гілка це посилання на коміт яка автоматично переміщується на останній коміт якщо ми його створюємо. Для того що б переміщатися між гілками виконується команда git checkout і назва гілки. Тоді head переміщується на останній коміт цієї гілки.

Навіщо створювати нову гілку ? Кожен розробник за допомогою гілок може створювати окремий функціонал і коли він буде готовий цей функціонал можна додати в основну гілку main шляхом злиття гілок

15. git branch <name branch> - команда для створення нової гілки. Ім'я гілки ліпше створювати виходячи з того функціоналу який ми створюємо

16. git checkout -b <name branch> - стоврення гілки та одразу перейти в цю гілку

17. git branch - показати список гілок

18. git branch -m <new name branch> - переіменувати гілку в якій ми зараз знаходимося

19. git branch -d <name branch> - видалити гілку але не можливо видалити гілку в якій ми зараз сидимо. Потрібно перейти в іншу і тоді можна видалити

(HEAD -> main, mew-branch) - head зараз знаходиться на перехрестті двох гілок

Коли ми об'єднуємо дві гілки то створюється коміт який має посилання на обідва останні коміта гілок. Такий коміт називається marge commit і він має 2 батьківських коміта 

Після того як ми злили гілки ту гілку яка була другорядна можна видалити

Якщо в різних гілках було змінено один і той самий файл то під час їх злиття може трапитися конфлікт. Це трапляється коли зміна була в одній і тій самій частині коду
